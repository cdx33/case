UX Case Study

BranchFlow

A branching interface for AI-assisted research.

Chat-style interfaces predate large language models by three decades. They were designed for transactional exchange, not for sustained intellectual work. BranchFlow is an MSc artefact testing whether interface design alone — not better models — can address five documented failure modes of LLMs used for research.

01

The gap chat doesn't close

Researchers, academic writers, and knowledge workers increasingly run early-stage thinking through LLMs — secondary research, exploratory writing, sensemaking. The interface they do it inside was never designed for that work. Five failure modes are now documented in the literature:

  1. Multi-turn drift Performance drops roughly 39% across generation tasks when models move from single- to multi-turn settings. When a model takes a wrong turn, it doesn't recover. (Laban et al., 2025)
  2. Context loss Earlier turns are forgotten as conversations grow. Quality degrades; the model repeats itself, then contradicts itself. Long-context windows help, they don't fix it. (Chhikara et al., 2025)
  3. Reduced cognitive depth Students using LLMs report lower cognitive load but produce lower-quality reasoning. Outsourcing the thinking thins the thinking. (Stadler, Bannert & Sailer, 2024)
  4. Metacognitive demands LLMs impose constant, unaddressed demands on the user — provenance, accuracy, confidence, integration — and the chat interface gives no support for any of them. (Tankelevitch et al., 2024)
  5. Loss of contextual richness AI summaries strip out historical trajectory, sociocultural framing, theoretical underpinnings — the very things that make scholarship scholarly.

BranchFlow directly addresses problems 1, 2 and 4 through architecture, and partly addresses 3 and 5 through interface affordances. What follows is the design bet, the artefact, and what testing did — and did not — confirm.

02

The design bet

Once learned, branching architecture and explicit commit semantics reduce sustained cognitive load by externalising state the user's working memory can't hold. That is the hypothesis. It rests on four design propositions.

i.

Architectural isolation

Exploration happens in branches, sandboxed from the confirmed knowledge base. If drift happens, it can't pollute upstream work.

ii.

Curated context

Only user-confirmed findings ship to the model. No full-history prompting, no growing context window, no implicit memory.

iii.

Explicit commit semantics

Promoting model output into the main set requires a deliberate user action. Adopting the AI's words is a choice, not a default.

iv.

Goal-relationship visibility

Research questions sit above findings; conclusions link back to them; the AI semantically assesses coverage. Goals stay visible while you work.

03

The artefact

BranchFlow is a single-page React app backed by a serverless LLM proxy. The working surface is three panels. The model on the right of every interaction is Gemini 2.5 Flash. The architecture, more than the UI, carries the design's argument.

Left panel: branches and subbranches — exploratory sandboxes off any finding. Centre panel: confirmed findings, grouped by topic, with research questions above and source upload alongside. Right panel: a detail surface that opens on click — references, scope tags, reasoning trace, and the option to challenge or branch from a finding.

The core loop is Finding → Branch → Conclusion. Findings are confirmed atoms of knowledge. Branches are sandboxes for investigating or challenging a finding. Conclusions synthesise multiple findings against a research question and are AI-routed to the question they best answer.

Two architectural commitments are encoded in code rather than copy, so they hold when the UI is redrawn:

Architectural commitment · I

The AI proposes, it never mutates

When the model wants to add, edit, merge, or archive a finding, it returns a [PROPOSED_ACTIONS] block alongside its prose. The proposal renders as an Apply / Dismiss card under the message.

Only the user's Apply click invokes the underlying handler. Apply / Dismiss is the structural answer to metacognitive monitoring: you don't have to police the AI, because it can't write to the record without you.

Architectural commitment · II

Only confirmed findings ship to the model

Drafts are invisible to the AI until you confirm them. Branches inherit the main knowledge base on entry, but their conversation history is sandboxed — branch chats don't leak into the main, and a branch shares its history only with its explicit children.

The result: a curated, user-authored memory layer instead of full-history prompting or external memory infrastructure.

On staleness. Editing one part of a project would, in a deterministic system, cascade through downstream branches. LLM outputs aren't deterministic, so re-running automatically would replace the user's interpretive work with a different roll of the dice. BranchFlow marks affected branches as stale instead, with a message describing what changed. The user decides whether to re-investigate. This is the central edit-propagation move: refuse to automate what only the user can judge.

04

How I tested it

Research-through-design (Zimmerman, Forlizzi & Evenson, 2007) across four phases: competitive analysis, concept design, prototype, evaluation. The prototype was supposed to be a Figma deck. It became a working React app on Vercel.

The design propositions hinge on how users react to real model behaviour — latency, refusal, partial coherence, stochastic phrasing. A static prototype would have produced cleaner data about a less honest object, so the prototype was extended into a working application integrated with Gemini 2.5 Flash via a Vercel serverless proxy. The cost was less time for evaluation.

Evaluation ran in two stages: one expert pilot, which served as a formative gate (it drove the terminology migration from node to finding and the introduction of the onboarding tour); then five unmoderated remote think-aloud sessions on the post-onboarding build. Four were valid. One was excluded for protocol non-compliance and kept as a methodological finding.

Mean SUS across n = 4 was 60.0 (range 45–80), below the 68-point industry mean (Bangor et al., 2008). With four participants and a novel interface in a single sub-hour session, SUS is framed here as a sanity check, not as evidence. The qualitative data carries the weight.

05

What surfaced

Reflexive thematic analysis (Braun & Clarke, 2019) across four transcripts produced thirty inductive codes, clustered through affinity mapping into seven themes. The strongest signal arrived early and unprompted.

4of4

participants spontaneously articulated the Finding · Branch · Conclusion model — without prompting, in their own words.

P2 called branches "pillars of the project." P1: "like a different file or notebook." P3: "a theory… until you actually test it." This is the design's largest conceptual achievement and the foundation everything else sits on. Beyond it, the themes split into five that travel beyond BranchFlow:

Conceptual clarity is not operational clarity

Major

Participants who understood the model perfectly could still get blocked at the verb surface. "Save," "Confirm," "Promote," "Suggest" — the analysis-to-main promotion path uses Suggest findings, which reads as redundant for findings already visible on screen.

"Why do I have to suggest another finding within a finding?" — P1

The transferable insight is the separability: concept-clarity work and verb-surface work are different design problems, and one can succeed while the other fails. The literature on AI interfaces does not currently treat this as its own dimension.

Trust runs through provenance — and refusal is part of the product

Major

Participants wanted a visible evidence chain: where did this claim come from? Citation metadata exists in branches but surfaces inconsistently across views. The counter-finding pushed back on a common product assumption:

"I like that the AI kept saying there was no data for my question." — P3

When the model refused rather than fabricating, trust increased. The conventional design assumption is that the AI should always answer; this finding inverts it. Refusal with rationale is a product feature, not a degradation.

Mental models can mismatch silently

Critical

The dataset task asked participants to bring a CSV into the project. The "Sources" affordance rejected it — CSVs route to "Analysis" instead. Three of four participants got stuck.

"I could not upload a CSV here." — P1

A Norman-style mapping failure: the user model says source for my research → Sources; the system model says CSV → Analysis. The generalisable implication: whenever the user and the system disagree about where something belongs, the interface needs a designed reconciliation path — a redirect, a tooltip, a route — not a silent rejection.

Onboarding lands the primitives, not the joins

Moderate

Cross-referencing the pilot (no onboarding) against the main study confirmed that the tour resolves first-contact terminology confusion — nobody in the main study asked what a Finding was. But the tour teaches what objects are, not how to stitch them together.

"I did not know how to integrate more knowledge to ask a different kind of question." — P3

Three separate tours — dashboard, new-project, project workspace — read as restarts rather than one continuous flow. Onboarding designs that teach primitives well still need to teach the joins between them, especially in tool surfaces with multiple modes.

The fifth participant was a method, not a person

Method

P5 skipped the onboarding tour and never opened the Analysis tab — both explicitly required by the brief. Their session was excluded from analysis but kept as a methodological finding: unmoderated remote testing cannot enforce protocol. A 1-in-5 non-compliance rate is not generalisable from this sample, but it's enough to over-recruit by at least 20% in any follow-up, and to add a brief synchronous orientation before the next study.

06

Design implications

Five generalisable lessons for AI research tools, in order of evidential confidence.

  1. The conceptual model is a primary design lever. Naming the canonical objects (Finding, Branch, Conclusion) clearly enough that users articulate the model in their own words is more valuable than any single feature. 4/4 evidence.
  2. AI refusal belongs in the product. Visible refusal with rationale builds trust faster than confident-sounding answers. The product question becomes: when should the model decline, and how does it show that — not just how do we minimise refusals?
  3. Treat conceptual clarity and operational clarity as separable problems. A user who understands what something is can still be stuck on what to click. Once primitives have landed, verb consistency across the product is a higher-leverage intervention than further conceptual elaboration.
  4. AI state must be visible at all times. When the model is silent, users assume the click failed and try again. Loading state, refusal state, citation state, source state — none of these can be optional. Silence reads as failure.
  5. Mental-model mismatches need designed reconciliation paths. When the user wants to put a CSV into "Sources" and the system routes CSVs to "Analysis," the interface needs to acknowledge the user's intent and redirect — not silently reject. This generalises far beyond file uploads.

07

What this study couldn't answer

The design's core cognitive claim — that branching plus commit semantics reduces sustained cognitive load — was not tested by this study, and could not have been. A single-session first-encounter evaluation can confirm that the conceptual model is learnable. It cannot distinguish germane schema acquisition from extraneous surface learning.

"Like studying for an exam to use it in practice." — P4

The principal next step is a two-week longitudinal A/B study (n ≥ 8) with a linear-chat baseline, NASA-TLX measured at first use and again after sustained use, within-participant design. Methodological infrastructure: a brief synchronous orientation to mitigate the P5 case, event instrumentation in the codebase to capture time-on-task and click paths, and a second coder for the thematic analysis. None of this is novel; the present study should have done it.

The most consequential learning, though, is methodological and not about the artefact at all: the design is what is being investigated, not the artefact. For research-through-design, interface decisions matter more than implementation polish, and the role of pilot feedback is to drive design iteration before code investment. The decision to extend the Figma prototype into a working React app was correct, but it was made late — and that timing cost evaluation depth. Were I starting again, the working prototype would exist before the first formative session, not after it.
